# Local System Test Plan

## Topology

Run two local permissioned-bank chains:

- Bank A: `npm run node:chainA`
- Bank B: `npm run node:chainB`

Deploy:

```bash
TMPDIR=/tmp XDG_CACHE_HOME=/tmp/.cache npm run deploy:multichain
```

Start workers:

```bash
npm run worker:hub
```

The worker hub starts:

- `checkpoint-relayer`: asks each source `BankCheckpointRegistry` to commit pending message ranges, gathers local bank-validator signatures over the emitted checkpoint object, and submits the certified checkpoint.
- `message-relayer`: builds Merkle inclusion proofs from the committed source message range and submits them.
- `risk-watcher`: observes pause/freeze/high-value policy state.

## Expected Lock to Mint

1. User locks collateral on Bank A.
2. UI shows message dispatched.
3. UI shows the message appended to the Bank A canonical message history.
4. Checkpoint worker commits a Bank A checkpoint on the source registry with message root, validator epoch, parent hash, message accumulator, and source block reference.
5. Bank A validators sign the exact source checkpoint hash.
6. Checkpoint worker submits the validator-certified Bank A checkpoint to Bank B.
7. UI shows the remote view certified and proof ready.
8. Message worker submits the Merkle proof.
9. Bank B consumes the message and mints wrapped collateral.
10. User deposits wrapped collateral and borrows stable token.

Expected result:

- wrapped collateral appears only after checkpoint and inclusion proof verification.
- relayer identity does not affect correctness.
- route policy can still block after proof verification, but cannot replace it.

## Expected Burn to Unlock

1. User repays or closes Bank B debt.
2. User withdraws wrapped collateral and burns it through `BridgeRouter.requestBurn`.
3. Bank B dispatches a release message.
4. Bank B checkpoint registry commits the release message into a source checkpoint.
5. Bank B validators certify the checkpoint.
6. Checkpoint worker submits the Bank B checkpoint to Bank A.
7. Message worker submits the inclusion proof.
8. Bank A consumes the message and unlocks original collateral.

Expected result:

- original collateral unlocks once and only once.
- replayed messages fail.

## Emergency and Policy Scenarios

- Set `PAUSE_ROUTE_ID=<route id>` before starting `risk-watcher` to pause a route.
- Set `FREEZE_ROUTE_ID=<route id>` before starting `risk-watcher` to freeze a route through secondary policy.
- Submit two valid but conflicting checkpoints for the same source sequence to trigger source freeze in `BankCheckpointClient`.
- Configure a low high-value threshold and verify that approval is required before execution.

## Local-Only Simplifications

- Validator signatures come from local dev-node accounts.
- Validator sets are modeled in `BankValidatorSetRegistry` and mirrored into each remote `BankCheckpointClient`.
- Checkpoint roots are generated by the source-chain `BankCheckpointRegistry` from on-chain `MessageBus` leaves.
- Source block references are local anchors, not full header verification.
- Checkpoint batch size is local and polling-dependent; multiple messages per checkpoint are supported.
- Recovery is represented by admin unfreeze and route policy functions.
