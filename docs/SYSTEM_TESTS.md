# Local System Test Plan

## Topology

Run two local permissioned-bank chains:

- Bank A: `npm run node:chainA`
- Bank B: `npm run node:chainB`

Deploy:

```bash
TMPDIR=/tmp XDG_CACHE_HOME=/tmp/.cache npm run deploy:multichain
```

Start workers:

```bash
npm run worker:hub
```

The worker hub starts:

- `checkpoint-relayer`: builds local finalized checkpoints, gathers local validator signatures, and submits them.
- `message-relayer`: submits message inclusion proofs.
- `risk-watcher`: observes pause/freeze/high-value policy state.

## Expected Lock to Mint

1. User locks collateral on Bank A.
2. UI shows message dispatched.
3. Checkpoint worker submits a Bank A checkpoint to Bank B.
4. UI shows checkpoint certified.
5. Message worker submits the Merkle proof.
6. Bank B consumes the message and mints wrapped collateral.
7. User deposits wrapped collateral and borrows stable token.

Expected result:

- wrapped collateral appears only after checkpoint and inclusion proof verification.
- relayer identity does not affect correctness.
- route policy can still block after proof verification.

## Expected Burn to Unlock

1. User repays or closes Bank B debt.
2. User withdraws wrapped collateral and burns it through `BridgeRouter.requestBurn`.
3. Bank B dispatches a release message.
4. Checkpoint worker submits a Bank B checkpoint to Bank A.
5. Message worker submits the inclusion proof.
6. Bank A consumes the message and unlocks original collateral.

Expected result:

- original collateral unlocks once and only once.
- replayed messages fail.

## Emergency and Policy Scenarios

- Set `PAUSE_ROUTE_ID=<route id>` before starting `risk-watcher` to pause a route.
- Set `FREEZE_ROUTE_ID=<route id>` before starting `risk-watcher` to freeze a route through secondary policy.
- Submit two valid but conflicting checkpoints for the same source sequence to trigger source freeze in `BankCheckpointClient`.
- Configure a low high-value threshold and verify that approval is required before execution.

## Local-Only Simplifications

- Validator signatures come from local dev-node accounts.
- Checkpoint roots are generated by scripts from local `MessageBus` events.
- The default worker commits one message leaf per checkpoint.
- Recovery is represented by admin unfreeze and route policy functions.
